## 잘된 것

1. **null 전파(Kleene 3치 논리)가 설계→구현→실데이터까지 일관** — 2.1이 washing_flag를 "하나라도 null→전체 null"에서 "확정 False 우선, 그 외 unknown→null"로 정교화했고, 드레스 리허설에서 실제로 `washing_flag: False 18 / None 8 / True 0`(소각 확정 종목은 진척률 몰라도 확정 False)로 설계대로 작동함이 실증됐다. 이론이 데이터로 검증된 드문 사례.
2. **grouping seam이 실제로 값을 했다** — 2.3이 `_build_populations(rows, group_of)`를 상수 그룹으로만 파둔 이음새 덕에, 2.7(sector peer-group)이 엔진 재작성 없이 KSIC 버킷 콜러블만 주입해 슬롯인됐다. "나중에 필요할 것 같은 추상화"가 아니라 리드가 미리 스코프를 분리(2.3 착수 전 결정)해서 만든 이음새라 실제로 맞아떨어졌다.
3. **API 패턴 재사용의 복리효과(1-2 어댑터 패턴과 동일 계열)** — 2.4(routers→services→repositories, Page 봉투, null-last 정렬)를 2.5가 그대로 복제(방향만 반전), 2.6은 outer join·sort 화이트리스트라는 새 요소가 있었지만 여전히 골격은 동일. 2.5 리뷰에서 나온 결함(빈 필터 확대·에러 계약 위반·page 오버플로)을 **2.6에 편승시켜 기존 valueup·metrics 라우터까지 패리티 정비**한 것도 같은 흐름.
4. **다층 리뷰가 계속 실질 버그를 잡음** — 2.1 High×5(AC3 위반·look-ahead·reconciliation 등), 2.3 High×2(mid-rank 백분위·macro null 대체), 2.5/2.6 Med 다수(sort 선검증 순서, 전용 예외 vs 광범위 ValueError). 매번 노이즈가 아니라 실제 계약 위반이었다 — 특히 2.6의 "pydantic ValidationError가 ValueError 하위 클래스라 내부 오류가 400으로 세탁된다"는 Python 특유의 미묘한 함정을 리뷰가 아니었으면 놓쳤을 것.
5. **드레스 리허설이 API 노출 전 마지막 방어선으로 작동** — 2.4~2.7 구현 후 실데이터 33종목을 완주시켜 배당 수집 공백(execution_score 0%)·금융주 섹터 문제(예측했던 것이 데이터로 확인)·파서 실전 성공률을 API가 공개되기 전에 발견. 이론적 defer 항목("finance 관점에서 문제일 것")이 "실제로 이만큼 문제"로 정량화됐다.

## 어려웠던 것

1. **2.1→2.3 아키텍처 전환 자체가 리스크** — 2.1은 corp별 독립 계산이었지만 2.3은 cross-sectional(전체 모집단 먼저, 그 다음 백분위)이라 근본적으로 다른 설계였다. 스토리 문서에 "이 스토리는 근본적으로 다르다"를 명시적으로 써서 재발명·오설계를 막았지만, 이런 전환점은 항상 위험 구간이다.
2. **HTTP 경계 결함이 패턴을 타고 전파** — 2.4에서 확립한 라우터 패턴을 2.5·2.6이 복제하면서, 2.5에서 처음 발견된 결함(빈 문자열 필터가 truthiness로 확대, page 무상한)이 **이미 2.4/metrics에도 존재했음이 뒤늦게 드러남**. 좋은 패턴 재사용의 이면 — 결함도 함께 복제되고, 발견도 늦게 온다.
3. **finance 도메인 지식과 데이터가 어긋난 지점을 미리 알았지만 실증까지는 기다려야 했음** — 2.3 착수 전 리드가 "은행 EV/EBITDA는 무의미하다"를 이미 알고 스코프를 분리(2-7 후속)했지만, 실제 금융주 7종목 전부 null이라는 정도는 드레스 리허설 전까지 확인 불가능했다. 도메인 직관과 데이터 실증 사이의 간극.
4. **사고 재발 방지가 완전하지 않음** — Epic 1 회고에서 이미 "확인 없는 삭제 금지"를 다루지 않았는데, 2.3 개발 중 `rm -f api.db`를 확인 없이 실행하는 사고가 재발(다행히 미참조 잔재 파일이라 무해). Epic 1에는 이 액션아이템이 아예 없었다 — 이번 회고에서 명시적으로 추가한다.
5. **execution_score의 실사용성이 현재 사실상 0%** — 배당 수집 공백(1.9로 이후 해소)이 드레스 리허설 시점 기준 achievement_rate 8%·execution_score 커버리지 심각. Epic 2의 API들은 정직하게 null을 노출하지만, "정직한 null"과 "쓸모있는 스크리너"는 다른 문제 — Epic 3 UI가 이 현실을 어떻게 다룰지 아직 미결정.

## 핵심 교훈

- **엄격 null(Kleene 3치)은 방어적 설계가 아니라 실사용 요구사항이었다** — "느슨하게 하나라도 있으면 평균 내기"였다면 드레스 리허설에서 워싱 오탐이 났을 것. 정직한 null이 스크리너의 신뢰도 그 자체.
- **seam은 다음 스토리가 실제로 쓸 때만 가치가 증명된다** — 2.3의 grouping seam은 "혹시 몰라서" 만든 게 아니라 리드가 명시적으로 스코프를 나눠서(2-7 후속 확정) 만들었고, 2.7이 실제로 그 형태 그대로 사용했다. 이음새를 팔 때는 "누가 언제 쓸지"를 확정해두는 것이 핵심.
- **패턴을 복제할 때 결함도 복제된다 → 새 스토리의 리뷰 결과를 과거 스토리에도 소급 점검할 것** — 2.5 리뷰 결과를 2.6에 편승시킨 것처럼, 앞으로도 "이 결함이 같은 패턴을 쓰는 다른 곳에도 있는가"를 리뷰 triage의 표준 질문으로 삼는다.
- **도메인 직관은 스코프 분리의 근거가 되지만, 정량화는 실데이터가 해야 한다** — "은행은 다를 것"이라는 판단은 착수 전 결정을 도왔지만, "7종목 전부 null"이라는 심각도는 드레스 리허설 없이는 몰랐다. 대형 계산 스토리 뒤에는 항상 실데이터 완주가 필요하다.
- **"확인 없는 삭제 금지"는 한 번 배우고 끝나는 교훈이 아니다** — Epic 1에서 발생했다면 Epic 1 회고 액션아이템에 올렸어야 했는데 누락됐다. 이번엔 명시적으로 액션아이템화.
