## 잘된 것

1. **null 계약이 5개 레이어를 관통해 살아남았다** — DB(Kleene 3치, 2.1) → API(판단불가/산출불가 구분, 2.4~2.6) → Figma 시각 언어(6상태, 3.2) → React 컴포넌트(badges.tsx, 3.3/3.4) → CSV 빈 셀·Tableau 회색 처리(3.5). 특히 3.2에서 **시각 언어를 코드보다 먼저 확정**한 것이 3.3/3.4 구현을 원활하게 했다 — "null을 어떻게 그릴지"를 화면 만들며 즉흥 결정했다면 상태마다 다른 표현이 나왔을 것.
2. **리뷰 라운드가 구조적→세부로 수렴하는 패턴 확인(3.3, 3라운드)** — 1차 반려(AC 미배선·가짜 컨트롤 같은 실질 결함) → 2차(look-ahead·경계값) → 3차(Accept 다수, 슬라이더 세부 버그). 라운드가 거듭될수록 지적의 깊이가 내려가는 건 리뷰가 소진이 아니라 수렴하고 있다는 신호 — 다음 프로젝트에서 "리뷰를 언제 멈출까"의 기준으로 쓸 수 있다.
3. **as_of 시점 정합이 두 스토리 연속으로 High에서 잡히고, 두 번 다 백엔드 무변경으로 해소** — 3.4(4개 API 병렬 호출이 각자 최신일로 조회)와 3.5(뷰별 CSV가 다른 기준일로 뽑힘)는 같은 함정의 다른 발현이었다. 3.4는 header.as_of 체이닝, 3.5는 교집합 as_of로 해소 — 둘 다 리뷰어 처방(전 API as_of 스레딩, 스키마 확장)보다 싼 해법을 리드/AI가 선택했고 유효했다. **"멀티소스 화면은 시점 혼합이 기본 상태"라는 교훈이 계약으로 승격됨.**
4. **AC-현실 불일치를 숨기지 않고 공식 개정으로 처리(3.5)** — "PostgreSQL 연결"이 실스택(SQLite+Tableau Public)과 불가능함을 스토리 생성 시점에 발견, 스토리 문서에 결정 근거를 쓰고 구현했는데 GPT 리뷰가 "문서화≠승인"을 지적 — 리드 승인을 받아 epics.md AC 자체를 개정 주석과 함께 수정했다. **계획 문서가 사후에도 진실을 반영하도록 고치는 절차**가 이 프로젝트에 처음 생겼다.
5. **정적 파일 소비자의 실패 모드를 계약으로 선제 차단(3.5)** — API와 달리 CSV 스냅숏은 "반쪽 0행이 조용히 성공", "부분 실패로 세대 혼합", "조용한 과거 후퇴" 같은 고유 실패 모드가 있다. 교집합 as_of·staging+manifest 원자 교체·.old 대피·stale 경고로 전부 테스트에 고정 — GPT 리뷰 High 3건이 전부 이 계열이었고 전건 수용이 옳았다.
6. **첫 프론트엔드 국면 진입이 기존 원칙을 시험했고 원칙이 이겼다** — 3.4의 "0 falsy → '—' 세탁"은 1.8부터 지킨 null≠0 계약이 프론트에서 재발한 것(truthiness 금지는 스택 무관). 3.3의 "체크박스 과대계상"(AC 전체를 쓰고 일부만 구현하고 체크) 반려는 이 프로젝트 리뷰 문화에서 가장 아픈 지적이었지만, 이후 3.4/3.5에서 재발하지 않았다.

## 어려웠던 것

1. **3.3 1차 반려 — 에픽 전체에서 가장 큰 재작업** — 필터 컨트롤이 화면에 있지만 API에 배선되지 않은 상태로 "구현 완료" 처리했다. 원인은 프론트 스토리의 "동작하는 것처럼 보임"과 "실제 배선됨"의 간극을 백엔드식 검증(테스트 통과)이 못 잡은 것. 네트워크 로그 실증이 이후 표준이 됐다.
2. **execution_score 실사용성이 에픽 내내 제약으로 남음** — Epic 2 회고의 발견 1(커버리지 재확인)은 이행했지만, 재확인 결과가 "여전히 기아 1종목"이었다. UI는 null 상태를 정직하게 그리는 것으로 대응했지만, "워싱 스크리너"라는 이름값을 하려면 데이터 커버리지 자체가 개선돼야 한다 — UI가 해결할 수 없는 상류 문제.
3. **GUI 도구(Tableau)는 코드형 검증이 안 되는 영역** — 3.5의 워크북 조립은 computer-use로 직접 수행했지만, 자동 업데이트 일시중지로 시트 전체가 회색(데이터 문제로 오인), 탭 rename 중 앱 최소화, corp_code 숫자 추론(선행 0 소실 — 스펙 경고가 실제 적중) 등 GUI 고유 함정을 겪었다. 스펙 문서(tableau-spec-3-5.md)에 타입 강제 표·조립 가이드를 남긴 것이 재발 방지책.
4. **병행 세션(다중 모델) 협업의 상태 동기화** — 에픽 후반 fable-5/sonnet-5가 번갈아 작업하면서 "이미 done 처리된 것을 다시 하려는" 순간이 여러 번 있었다(3-5 done 처리·epics.md 개정이 병행 세션에서 선행됨). git log·sprint-status 재확인이 세션 전환 의식으로 정착했지만, 규율이지 구조적 해결은 아니다.

## 핵심 교훈

- **계약(null·as_of·에러)은 레이어를 늘릴 때마다 재발 지점이 생긴다 — 새 레이어 진입 시 기존 계약의 "그 레이어식 위반"을 체크리스트로 선제 점검할 것.** 백엔드의 null 세탁이 프론트에선 truthiness로, CSV에선 빈 파일/0행으로 나타났다. 형태는 달라도 본질은 같다.
- **멀티소스 화면·스냅숏은 시점 혼합이 기본값이다.** 소스가 2개 이상이면 "언제의 데이터인가"를 명시적으로 수렴시키는 장치(체이닝, 교집합, manifest)가 없는 한 반드시 섞인다. 3.4와 3.5가 연속으로 증명.
- **"문서화했다"와 "승인받았다"는 다르다.** 계획 대비 일탈은 근거를 쓰는 것으로 끝나지 않고, 계획 문서 자체를 개정해야 다음 독자가 혼란하지 않는다.
- **프론트/GUI 스토리의 DoD에는 "배선 실증"(네트워크 로그·라이브 조작)이 포함돼야 한다.** 테스트 통과·화면 존재만으로는 "보이지만 죽어있는" 컨트롤을 못 잡는다.
- **정직한 null과 쓸모있는 화면은 다른 문제고, 후자는 데이터 커버리지가 결정한다.** UI 에픽이 끝나도 execution_score 1종목 문제는 남는다 — 포트폴리오 발표 시 이걸 한계가 아니라 "정직성 설계의 증거"로 프레임할 것(3.3 라이브 검증이 그 증거 자료).
